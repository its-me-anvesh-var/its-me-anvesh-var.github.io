DPDP Act 2023 — What Indian Startups Actually Need To Do Right Now
→

The Digital Personal Data Protection (DPDP) Act 2023 is now in effect, and Indian startups — especially in fintech, edtech, and healthtech — need to understand what compliance actually looks like.

Here's what you MUST do right now:

1. Audit Your Data Flows
Map what personal data you collect, where it's stored, who has access, and why you need it. Most startups fail here because they've never documented their data lifecycle.

2. Implement Consent Mechanisms
Users must give explicit, informed consent before you collect their data. Pre-ticked boxes don't count. Your consent flow must be clear, specific, and revocable.

3. Appoint a Data Protection Officer (if required)
If you process significant volumes of data, you'll need a designated DPO. Even if not mandated, having someone accountable is good practice.

4. Update Your Privacy Policy
Your privacy policy must now explain:
• What data you collect and why
• How long you retain it
• How users can access, correct, or delete their data
• Your breach notification process

5. Build a Breach Response Plan
DPDP requires breach notification to the Data Protection Board (DPBI) and affected users. You need a documented incident response process — now, not when a breach happens.

6. Review Third-Party Vendors
Every SaaS tool, payment gateway, and analytics platform you use is a data processor. Ensure they're DPDP-compliant or you're liable for their violations.

Penalties? Up to ₹250 crore for violations. The Act gives DPBI wide enforcement powers.

Timeline? Rules are still being drafted, but don't wait. Start with basics: data mapping, consent flows, and breach response. These take months to implement properly.

5 Splunk Detection Rules Every SOC Analyst Should Have
→

After building detection rules for SOC environments, these 5 Splunk queries are non-negotiable. They catch the most common attack patterns I've seen in real incidents.

1. Brute Force Detection (RDP + SSH)

index=main sourcetype=WinEventLog:Security EventCode=4625
| stats count by src_ip, user, host
| where count > 10
| eval severity="HIGH"

Why: Brute force is still the #1 initial access method. 10+ failed logins from one IP in a short window = investigate immediately.

2. After-Hours Admin Login

index=main sourcetype=WinEventLog:Security EventCode=4624
| eval hour=strftime(_time, "%H")
| where (hour < 8 OR hour > 20) AND match(user, "(?i)admin")
| table _time, user, src_ip, host

Why: Legitimate admins rarely work at 3 AM. This catches compromised accounts and insider threats.

3. PowerShell Download Cradle

index=main sourcetype=WinEventLog:Security EventCode=4688
  New_Process_Name="*powershell*"
| where match(Process_Command_Line, 
    "(?i)invoke-webrequest|iwr|downloadstring")
| table _time, user, Process_Command_Line, host

Why: 90% of malware droppers use PowerShell to fetch payloads. Catches Emotet, Cobalt Strike, and most ransomware.

4. Large Data Exfiltration

index=network sourcetype=firewall_logs
| stats sum(bytes_out) as total_bytes by src_ip, dest_ip
| where total_bytes > 104857600
| eval total_mb=round(total_bytes/1024/1024, 2)

Why: Data theft happens in bulk. 100MB+ transfers to external IPs = investigate the destination and what was sent.

5. New Local Admin Account Created

index=main sourcetype=WinEventLog:Security EventCode=4720
| table _time, user, TargetUserName, host
| eval severity="CRITICAL"

Why: Attackers create backdoor admin accounts for persistence. Every new account creation should be verified.

Pro Tips:
• Set these to run every 5 minutes
• Alert on High/Critical only to avoid fatigue
• Map each to MITRE ATT&CK techniques
• Test with attack simulations (use Atomic Red Team)

These 5 rules cover: Initial Access (T1110), Execution (T1059), Persistence (T1136), and Exfiltration (T1041). That's 80% of real-world incidents.

From Student to SOC Analyst — Honest Roadmap for Indian Freshers
→

Most "SOC Analyst roadmaps" are written by people who've never worked in a SOC. Here's what actually works in India's job market.

Reality Check First:
• Entry-level SOC roles pay ₹3.5–6 LPA in tier-1 cities
• You'll start as L1 (alert triage, not threat hunting)
• 80% of the job is following playbooks, not hacking
• Night shifts are common in the first 1–2 years

The Actual Roadmap:

Phase 1: Foundations (3 months)
✅ Networking basics: TCP/IP, DNS, HTTP/S, ports
✅ Windows + Linux fundamentals (not advanced)
✅ Learn how to read logs (Windows Event Logs, Syslog)
✅ Understand MITRE ATT&CK framework
Free resource: Professor Messer's Security+ course on YouTube

Phase 2: Core Skills (3–4 months)
✅ Get CompTIA Security+ (₹25,000 exam, worth it)
✅ Learn one SIEM: Splunk (free trial) or Wazuh (open source)
✅ Build a home lab (see my post on this)
✅ Write 5–10 detection rules mapped to MITRE
GitHub matters: Document everything you build

Phase 3: Hands-On Practice (2–3 months)
✅ TryHackMe: SOC Level 1 path
✅ LetsDefend: Real SOC scenarios
✅ CyberDefenders: Blue team CTFs
✅ Practice writing incident reports
Pro tip: Employers ask "walk me through how you'd investigate a phishing alert" — have an answer ready

Phase 4: Apply (Ongoing)
Target companies:
• MSSPs: Tata TCS, Accenture SOCs, IBM X-Force
• Banks: HDFC, ICICI, SBI (have internal SOCs)
• Product companies: Zscaler, Palo Alto, Crowdstrike India
• Startups: Often hire L1 analysts, pay similar to MNCs

Resume Tips:
❌ Don't write: "Knowledge of SIEM tools"
✅ Write: "Built Wazuh SIEM with 15 custom detection rules covering MITRE T1110, T1566, T1059"

Common Mistakes:
• Chasing CEH before Security+ (CEH is expensive, not needed for L1)
• Focusing on penetration testing (that's offensive, SOC is defensive)
• Not documenting your lab work on GitHub
• Waiting for "the perfect time" to start (start messy, improve later)

Timeline: 8–12 months from zero to first SOC job if you're consistent. Don't rush certifications — build real skills.
